# Build Server

This directory is used to simulate the build server, which is responsible for compiling source code and generating raw binary files (no private keys or signatures).

## Overview

The build server compiles the Armino project and generates binary files that will be used by the sign server and pack server. It does NOT perform any signing operations - all binaries generated are unsigned.

## How to Run

### Direct Execution

The build server is typically invoked through the main `run.py` script with the `--build` option:

```bash
python3 -B run.py --build --project security/xip [other options...]
```

### Standalone Execution

You can also run the build server directly:

```bash
cd build_server
python3 build.py security/xip
```

Or manually execute the build commands:

```bash
# Navigate to Armino root directory (5 levels up from examples/flash_all_app)
cd ../../../../

# Clean previous build
make clean

# Build the project
make bk7236n PROJECT=security/xip APP_VERSION="vvvvvvvvvvvvvvvvvvvvvvvv1.0"
```

## Build Process

The build server performs the following steps:

1. **Clean previous build**: Executes `make clean` to remove previous build artifacts
2. **Compile project**: Executes `make bk7236n PROJECT={project} APP_VERSION="vvvvvvvvvvvvvvvvvvvvvvvv1.0"`

### Build Command Details

```bash
make bk7236n PROJECT={project} APP_VERSION="vvvvvvvvvvvvvvvvvvvvvvvv1.0"
```

- **SOC**: `bk7236n` (hardcoded in build.py)
- **PROJECT**: Project name (e.g., `security/xip`, `security/secureboot_no_tfm`)
- **APP_VERSION**: Application version string (currently hardcoded as `"vvvvvvvvvvvvvvvvvvvvvvvv1.0"`)

**Note**: The APP_VERSION in build.py is hardcoded. The actual version used for signing is passed via `--app_version` parameter to `run.py`, which is used by the sign server.

## Output Files

The build server generates files in the Armino build output directory structure:

### Main Build Output Directory

**Location**: `build/bk7236n/xip/package/`

This directory contains the packaged binaries and configuration files:

#### Binary Files

1. **bl2.bin** (Required)
   - Description: MCUBoot bootloader binary
   - Used by: Sign server (if BL1 secure boot enabled), Pack server
   - Location: `build/bk7236n/xip/package/bl2.bin`

2. **primary_all.bin** (Conditional - generated in non-XIP mode)
   - Description: Primary application binary (combined from child partitions)
   - Used by: Sign server (in OVERWRITE mode)
   - Location: `build/bk7236n/xip/package/primary_all.bin`
   - **Note**: In XIP mode, `primary_all.bin` is generated by sign server from `app.bin`

3. **xip_a.bin** (Required)
   - Description: XIP application binary (original, before signing)
   - Used by: Pack server for creating `app_signed.bin`
   - Location: `build/bk7236n/xip/package/xip_a.bin`

4. **user_mfr.bin** (Required)
   - Description: User manufacturing binary
   - Used by: Pack server
   - Location: `build/bk7236n/xip/package/user_mfr.bin`

5. **partition.bin** (Required)
   - Description: Partition table binary
   - Used by: Pack server
   - Location: `build/bk7236n/xip/package/partition.bin`

6. **provision.bin** (Conditional - only when BL1 secure boot enabled)
   - Description: Provisioning binary for BL1 secure boot
   - Used by: Pack server
   - Location: `build/bk7236n/xip/package/provision.bin`

#### Configuration Files

7. **partitions.csv** (Required)
   - Description: Partition table configuration
   - Used by: Sign server (for XIP mode to generate primary_all.bin), Pack server
   - Location: `build/bk7236n/xip/package/partitions.csv`

8. **pack.json** (Required)
   - Description: Packing configuration file
   - Used by: Pack server
   - Location: `build/bk7236n/xip/package/pack.json`

9. **nvs.csv** (Required)
   - Description: NVS (Non-Volatile Storage) configuration
   - Used by: Pack server
   - Location: `build/bk7236n/xip/package/nvs.csv`

10. **primary_manifest.json** (Conditional - only when BL1 secure boot enabled)
    - Description: BL1 manifest JSON configuration
    - Used by: Pack server
    - Location: `build/bk7236n/xip/package/primary_manifest.json`

11. **root_ec256_pubkey.pem** (May be generated)
    - Description: Root public key PEM file
    - Used by: Pack server
    - Location: `build/bk7236n/xip/package/root_ec256_pubkey.pem`

### Application Binary Directory

**Location**: `build/bk7236n/xip/bk7236n/`

This directory contains the compiled application binary:

1. **app.bin** (Required for XIP mode)
   - Description: Compiled application binary (unsigned)
   - Used by: Sign server (in XIP mode to generate primary_all.bin)
   - Location: `build/bk7236n/xip/bk7236n/app.bin`
   - **Note**: This file is copied to sign_server when `ota_type == 'XIP'` or `bl1_secureboot_en == True`

## File Transfer to Other Servers

After build completes, the main `run.py` script copies files to other servers:

### Files Copied to Sign Server

1. **app.bin** (Conditional)
   - Source: `build/bk7236n/xip/bk7236n/app.bin`
   - Destination: `sign_server/app.bin`
   - Condition: When `ota_type == 'XIP'` or `bl1_secureboot_en == True`
   - Used for: Generating `primary_all.bin` in XIP mode

2. **bl2.bin** (Conditional)
   - Source: `build/bk7236n/xip/package/bl2.bin`
   - Destination: `sign_server/bl2.bin`
   - Condition: When `bl1_secureboot_en == True`
   - Used for: Generating `primary_manifest.bin`

### Files Copied to Pack Server (immutable directory)

All files from `build/bk7236n/xip/package/` are copied to `pack_server/immutable/`:

- `bl2.bin`
- `user_mfr.bin`
- `nvs.csv`
- `pack.json`
- `partitions.csv`
- `partition.bin`
- `xip_a.bin`
- `root_ec256_pubkey.pem`
- `provision.bin` (if BL1 secure boot enabled)
- `primary_manifest.json` (if BL1 secure boot enabled)

## Important Notes

### 1. Build Directory Location

The build server must be executed from the Armino root directory (5 levels up from `examples/flash_all_app/`). The `run.py` script automatically changes to this directory when `--build` is specified.

### 2. APP_VERSION Parameter

- The `APP_VERSION` in `build.py` is hardcoded as `"vvvvvvvvvvvvvvvvvvvvvvvv1.0"`
- The actual version used for signing is specified via `--app_version` parameter to `run.py`
- The build server's APP_VERSION is used during compilation (as a compile-time definition)
- The sign server uses the `--app_version` parameter for signing operations

### 3. XIP Mode vs OVERWRITE Mode

- **XIP Mode**: 
  - `primary_all.bin` is NOT generated by build server
  - `app.bin` is copied to sign_server
  - Sign server generates `primary_all.bin` from `app.bin` before signing
  
- **OVERWRITE Mode**:
  - `primary_all.bin` is generated by build server (from child partitions)
  - `app.bin` is NOT copied to sign_server
  - Sign server uses the pre-generated `primary_all.bin`

### 4. Clean Build

The build server always executes `make clean` before building. This ensures a clean build environment but may increase build time.

### 5. Project Path

The project path should be relative to `projects/` directory in Armino:
- Example: `security/xip` → `projects/security/xip`
- Example: `security/secureboot_no_tfm` → `projects/security/secureboot_no_tfm`

### 6. Build Time

The build process can take several minutes depending on:
- Project complexity
- Number of components
- Whether it's a clean build or incremental build
- System performance

## Troubleshooting

### Build Fails

1. **Check project path**: Ensure the project exists in `projects/` directory
2. **Check dependencies**: Ensure all required tools and dependencies are installed
3. **Check disk space**: Ensure sufficient disk space in build directory
4. **Check logs**: Review build output for specific error messages

### Files Not Generated

1. **Check build output**: Verify files exist in `build/bk7236n/xip/package/`
2. **Check project configuration**: Ensure project is configured correctly
3. **Check OTA type**: Some files are only generated for specific OTA types

### APP_VERSION Mismatch

- The APP_VERSION in build output may differ from the version used for signing
- This is expected: build uses hardcoded version, signing uses `--app_version` parameter
- The signing version is the one that matters for firmware identification
